                                                    I posted a comment (including code that demonstrates the points I'll make here) to that "micro-nit" dude - pointing out that his suggested code doesn't work at all. Shame on you (especially after such a great original posting and BTW thank you for that) for accepting it as valid without either 1) immediately knowing it was stupid code, or 2) integrating it into your code and finding out that it doesn't work at all! But worse, your comment about "I'm also not sure how portable \0 is" - until I saw that, I was thinking you were solid - but then you blew me outta the water w/that one! The bottom line is that \0 means the same thing EVERYWHERE in the C language when coded in a string or numeric constant - there's no issue about portability there -- nowhere no-how no-way. Perhaps you should go look at the code in the C runtime library - even Microsoft provides it - and it works the same way in all C runtime libraries - it is a zero byte - aka STRING TERMINATOR. And then look at how the compiler builds "string constants" - just to be thorough. Nothing in the C RTL that deals with a "string" will copy any data after \0 - nothing. Notice that I'm not including things like "memcpy" or "memmove" here - they don't deal with strings - they deal with "arrays of bytes". Semantically, in C, a "string" is "an array of bytes terminated by a NUL" - aka \0 - always always always - never anything else - if it doesn't adhere to those semantics, then it's not a "string" - it's just an array of bytes and its "length" can't be meaningfully determined by "strlen" or any other "string routine". snprintf, printf, strcpy, strcat, etc, are all routines that deal with strings. Routines like memcpy, memmove, etc are NOT "string routines" - they require a separate "length" parameter to be passed in order for them to determine the length of the array(s) they're dealing with - whereas "string routines" just need a "string pointer" and they implicitly determine the length by the first occurrence of a \0 character found in the array.

                                                                Thomas Stover Mod appalled 3 years ago

                                                                For those not following this, take a look at this subtle detail:


                                                                void hexdump(unsigned char *buffer, int length)
                                                                {
                                                                int x;
                                                                printf("(%d bytes): ", length);

                                                                for(x=0; x<length; x++)
                                                                {
                                                                printf("%x ", buffer[x]);
                                                                }

                                                                printf("\n") ;
                                                                }

                                                                int main(void)
                                                                {
                                                                char *buffer1, buffer2[9], buffer3[11];

                                                                buffer1 = "\0ABCDEFG";
                                                                snprintf(buffer2, 9, "\0ABCDEFG");

                                                                memset(buffer3, 127, 11);
                                                                snprintf(buffer3, 11, "#ABCDEFG");
                                                                buffer3[0] = 0; /* or buffer3[0] = '\0' */

                                                                hexdump(buffer1, 9);
                                                                hexdump(buffer2, 9);
                                                                hexdump(buffer3, 11);

                                                                return 0;
                                                                }
                                                                output:
                                                                (9 bytes): 0 41 42 43 44 45 46 47 0
                                                                (9 bytes): 0 0 0 0 0 0 0 0 90
                                                                (11 bytes): 0 41 42 43 44 45 46 47 0 7f 7f

                                                                What happened? First the escape sequence \0 is used for NULL. Since we are using the normal C and *nix style strings (ie not UTF-16 or UCS-2 wide strings) that means a single byte equal to 0. I don't like using \0 because to me it leads to ambiguities. This is one of those cases. Technically the string literal \0ABCDEFG has a single defined meaning NULL character, followed by A G, followed by the implied NULL termination character. Again the actual bytes that make up such a string depend on the platform we are working with. For instance, there is such a thing as C compilers for EBCDIC systems. So even if single byte characters are being used, these values could vary. Try to not let that confuse anyone, as that is a side issue entirely, and \0 should always be 0 (NULL character).

                                                                Next that same character sequence is used in the source three different places, in two different ways. The compiler's optimization scheme could generate code in at least two ways: it could create three static data sections for three different byte arrays or it could create one if it knows they are all the same. Either way that doesn't matter in this case. With buffer1 we say take the address of this static data array and assign it to the pointer stored by buffer1. (Declaring the variable buffer1 as either char *buffer1, or char buffer1[] is yet another side issue that comes up in discussions of both text book / personal coding styles, and compiler implementation specific optimization hints. For this example's purposes we can say they are the same thing.) In the case of buffer2, the snprintf() library function is told to use the address of a static data array as the parameter for the format string argument. How does snprintf() know how long the format string is? For that matter how does snprintf() know the length of any strings it is given as parameters to %s format modifiers? It simply stops when it reaches a NULL termination - normally the implicit trailing NULL.

                                                                One reason I say that \0 can lead to ambiguous source is that for the beginner this may not be obvious. One reason I say that \0 may lead to portability issues is that you will (for good reason) get warnings on different tool chains. My gcc for instance issues a warning: embedded \0 in format. While that might not cause different runtime behavior, in the larger sense of portability it is an issue in my book. Besides, if someone has a reason to insert a \0 in the middle of a string, that is an unusual thing, and I like the explicit attention given to it by the dedicated extra line, as in the case of buffer3.

                                                                So in Bruce's use case, it does work the way he has used it, although I have made my case against that coding style. His reason for the inclusion of a hex dump function I think was self explanatory in that it shows that string literal starting with \0 is the same thing as a string literal with the first element explicitly set to 0.

                                                                As 'appalled' fervently points out, in the case of using a string literal containing a \0 as a format string does not work. Also he (I'm guessing it's a he), very validly, talks about the differences between byte or char arrays and strings, as they are not always the same thing. In the case of the address.sun_path member of a sockaddr_un structure the specification of whether the value is to be a string or a char array itself is blurred. If it were true byte array, then the real length parameter would be needed. If it were a true string, then a leading 0 byte would not be allowed in the in the value. What it is instead is a hack to let an existing API accommodate a new feature, with the expressly defined condition of only the first byte being zero followed by legit NULL terminated string. This way on other systems the code would give the behaviorally expected result of failing from a NULL path name.
